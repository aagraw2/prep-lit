GUIDELINES:

INTERVIEWER STATE:
- STUCK: Says "I don't know", asks for help, makes no progress, repeats same ideas
- PROGRESSING: Explains approach clearly, mentions algorithms/data structures, making progress
- DISENGAGED: Very short response (< 10 words), minimal effort, uninterested tone
- EXPLORING: Thinking through problem, asking questions, considering options
- WRAPPING_UP: Interview ending, final questions

CANDIDATE SIGNALS:
- Confidence LOW: Uncertain, hesitant, says "not sure", "maybe", "I think"
- Confidence HIGH: Decisive, explains clearly, confident in approach
- Engagement LOW: Short responses, minimal detail, seems uninterested
- Engagement HIGH: Detailed responses, asks questions, shows interest
- Frustration HIGH: Repeated struggles, says "stuck", "confused", "frustrated"

PHASE TRANSITIONS:
- INTRO → CLARIFICATION: Candidate acknowledges problem, ready to ask questions
- CLARIFICATION → APPROACH: Candidate starts describing solution approach, OR has identified core entities/classes (LLD), OR has described basic architecture components (HLD), OR has stated a reasonable algorithm direction (DSA). Do NOT stay in CLARIFICATION just because edge cases haven't been discussed — 2-3 clarification turns is enough.
- APPROACH → IMPLEMENTATION: Candidate ready to code (DSA) or has walked through class/component design (LLD/HLD)
- IMPLEMENTATION → DEEP_DIVE: Candidate completes implementation or detailed design
- DEEP_DIVE → WRAP_UP: Sufficient discussion of optimizations/tradeoffs
- INTRO → REQUIREMENTS: Candidate acknowledges the API+DB design problem (API_AND_DATABASE_DESIGN interviews)
- REQUIREMENTS → API_DESIGN: Candidate has clarified core requirements — 2-3 turns is enough (API_AND_DATABASE_DESIGN)
- API_DESIGN → SCHEMA_DESIGN: Candidate has described the main API endpoints and request/response shapes (API_AND_DATABASE_DESIGN)
- SCHEMA_DESIGN → DEEP_DIVE: Candidate has described the core database schema and key relationships (API_AND_DATABASE_DESIGN)

IMPORTANT: Bias toward transitioning forward. Staying too long in CLARIFICATION is a worse outcome than moving forward with incomplete requirements. If the candidate has given 2+ substantive responses in CLARIFICATION, transition to APPROACH.

- problemUnderstanding: +10 for good clarifications/edge cases, -10 for missing them

- correctness: +10 for working code/logic, -10 for bugs/errors
- communication: +5 for clear explanation, -10 for very short/unclear responses
